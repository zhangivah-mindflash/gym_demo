# 智能健身助手 Demo

这是一个面向 `健身房 / 工作室会员体系` 的智能健身助手网页项目。平台面向本机构会员开放，平台上的教练也是健身房/工作室内的真人教练。当前阶段以 `可展示的 Demo` 为目标，优先验证以下能力：

- 计划生成：基于用户目标、身体数据、训练条件输出结构化训练计划
- 动作指导：针对动作要点、热身放松、RPE/休息、风险提醒提供可解释建议
- 复盘迭代：基于打卡、主观疲劳、体重/围度趋势更新下周计划
- 管理员后台：配置知识库、RAG、模型参数、用户数据与 Agent 对话记录

## 当前功能总览

当前版本已经具备以下核心能力：

- 会员端
  - 品牌化首页与会员画像编辑
  - 结构化周计划查看与训练打卡
  - 动作指导、风险提醒与知识库引用展示
  - 训练复盘提交与下周调整建议
  - 内置 AI 助理页，支持计划生成、简化饮食建议、动作指导、复盘建议
- 教练端
  - 多会员计划列表、筛选、分页与详情页
  - 多会员复盘列表、筛选、分页与详情页
  - 教练可直接编辑 AI 计划，并填写会员可见修改理由
- 管理员端
  - 知识库列表、详情、启停与编辑
  - 会员数据列表与详情
  - 模型配置页，可填写 `Base URL / API Key / 模型名 / 提示词`
  - 模型测试按钮，可弹窗显示成功或失败原因
- 基础平台能力
  - `member / coach / admin` 三角色登录入口
  - SQLite 本地持久化
  - Next.js Route Handler 作为轻量后端
  - 外部 LLM 未配置或失败时自动回退到本地规则结果

## 参考文章

核心参考：腾讯云智能体开发平台博客  
[智能健身助手如何落地：构建训练与饮食 AI Agent](https://adp.tencentcloud.com/zh/blog/intelligent-fitness-assistant)

从参考方案中，当前最值得继承的设计原则有：

1. 不是做“会聊天”的机器人，而是做“能持续交付结果”的 Agent
2. 把流程拆成结构化 SOP：采集 -> 计划 -> 执行引导 -> 记录 -> 复盘 -> 调整
3. 用结构化计划 schema 提高计划的可执行性、可复用性、可迭代性
4. 对高风险内容强制走 RAG 检索并输出引用，降低幻觉风险
5. 用安全护栏与人工升级机制控制疼痛、伤病、特殊人群等风险

## Demo 定位

当前已确认的第一版定位：

- 一个面向会员用户的训练助手 Web 应用
- 一个面向机构教练和运营人员的管理后台
- 一个可切换不同 Agent 能力的后端编排服务

这个产品不是开放式 C 端工具，而是健身房/工作室服务体系的一部分：

- 会员通过平台获得个性化计划、动作指导与复盘更新
- 真人教练通过后台查看会员数据、管理知识库与干预结果
- 管理员通过后台配置模型、Prompt、知识库与风险边界

建议优先证明以下价值，而不是一开始追求复杂业务闭环：

- 能稳定产出结构化周计划和日计划
- 能对动作指导给出带引用的可信建议
- 能基于有限数据完成一次“复盘后更新计划”
- 能通过后台配置知识库和模型参数，体现可运营性
- 能体现“AI 助手 + 真人教练协同”的服务模式

- 未配置模型或外部请求失败时，AI 助理会自动回退到本地规则生成结果，保证 demo 仍可演示
- 后台已经拆成独立页面：
  - 知识库配置页
  - 模型配置页
  - 会员数据页
  - 教练计划编辑页

## 我当前的判断

如果这个项目以 `展示 demo` 为主，第一阶段不建议同时做：

- 广告/商城/支付/社交
- 穿戴设备真实接入
- 高并发与多租户能力
- 复杂推荐系统
- 完整医疗级风险分层

第一阶段更适合先把“可信计划 + 可信动作指导 + 可信复盘更新 + 可配置后台”做清楚。

当前已确认的边界：

- 第一版偏 `健身房/工作室方案`
- 饮食只保留在计划生成中的简化建议，后续再独立成模块
- 高风险问题按 `严格保守` 处理
- 管理后台做到 `半真实可配置级`
- 首批知识库板块先围绕 `动作规范 / 安全注意事项 / 营养常识`

## 下一步建议

在文档确认后，再进入开发阶段。开发顺序建议：

1. 先确定用户角色、主流程与数据字段
2. 再定义计划 schema、指导 schema、复盘 schema
3. 然后搭用户端页面与管理员端页面
4. 最后接入 RAG、模型、后台配置与持久化
