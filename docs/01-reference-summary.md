# 参考方案总结

参考来源：  
[腾讯云智能体开发平台博客：智能健身助手如何落地：构建训练与饮食 AI Agent](https://adp.tencentcloud.com/zh/blog/intelligent-fitness-assistant)

## 文章核心观点

文章强调，智能健身助手的关键不在于“像教练一样聊天”，而在于是否能把用户带到结果。落地重点不是单次问答，而是形成一个可运营、可治理、可度量的闭环。

文章给出的核心链路是：

`采集 -> 计划 -> 执行引导 -> 记录 -> 复盘 -> 调整`

这和你的设想高度一致，尤其契合下面三个场景：

- 计划生成
- 动作指导
- 复盘迭代

## 对我们最有价值的设计要点

### 1. 用结构化 SOP 替代自由聊天

参考文章建议把对话拆成结构化服务流程，而不是让模型自由发挥。  
这对我们的 Demo 很重要，因为：

- 能降低回答飘忽和前后不一致
- 更容易把结果渲染成卡片、表格、时间线
- 更方便保存到用户个人页面并持续迭代

## 2. 意图路由与对话管理必须前置

文章明确提到，需要识别并分流不同意图，例如：

- 计划生成
- 动作指导
- 饮食建议
- 复盘调整

对我们的 MVP 来说，建议至少先做三类路由：

1. `plan_generation`
2. `movement_guidance`
3. `review_iteration`

并且维护一个最小会话状态：

- 当前用户画像是否完整
- 当前正在处理哪个任务
- 当前是否存在风险信号
- 当前计划版本号

## 3. 结构化计划 schema 是第一优先级

文章提出的优化点里，最适合直接借鉴的是 `计划 schema`。

原因很直接：

- 自然语言计划很难存储
- 很难复盘
- 很难做版本更新
- 很难在页面中稳定展示

建议我们的计划至少拆成两层：

1. 周结构：周期目标、每周训练频次、训练日分配、阶段重点
2. 日计划：动作、组数、次数、RPE、休息、替代动作、注意事项

## 4. 高风险内容强制走 RAG

文章特别强调，对以下内容要强制检索并带引用输出：

- 动作要点
- 禁忌动作
- 疼痛处理边界
- 风险提醒

这对“动作指导”模块尤其关键。建议把“可信输出”做成规则，而不是可选项：

- 普通计划解释可以走模型生成
- 高风险动作指导必须先检索再生成
- 出现疼痛、眩晕、伤病史、孕期、未成年人等信号时，触发更严格的提示或限制

## 5. 护栏与升级机制不能后补

文章给出的一个重要结论是：  
在健康与运动场景里，边界控制不能作为后期补丁，必须一开始就设计。

建议最低限度包含：

- 不替代医疗建议
- 对伤病、剧烈疼痛、头晕、胸闷等内容触发红线提示
- 对特殊人群给出额外限制
- 必要时建议转人工教练或线下专业机构

## 6. 先做轻量闭环，不依赖穿戴设备

文章明确提到，没有穿戴设备数据也能做闭环。  
第一阶段完全可以用以下数据代替：

- 用户打卡
- 主观疲劳
- 体重趋势
- 围度变化
- 完成率

这正适合当前 demo 目标，也能避免前期集成成本。

## 适合我们直接采用的落地原则

结合参考文章与当前目标，我建议直接采用下面这套原则：

1. `先结构化，后智能化`：先把输入输出 schema 定义清楚，再接复杂模型能力
2. `先闭环，后扩展`：先打通计划、指导、复盘三段主链路
3. `先可信，后丰富`：高风险内容必须有检索与引用
4. `先可展示，后工程化`：先做一个能完整演示的产品，再谈复杂扩展

## 对 Demo 的直接启发

如果我们要做一个能展示给客户、团队或投资人的 Demo，那么最应该展示的不是“模型很聪明”，而是下面四点：

1. 能采集完整画像并形成个性化约束
2. 能输出可执行、可保存、可更新的结构化计划
3. 能对动作指导给出带引用的安全建议
4. 能根据记录和反馈自动给出下一轮调整

这四点比“多轮闲聊效果很好”更能体现产品价值。
