# 已确认决策与后续问题

这份文档用于记录已经确认的关键决策，并保留少量后续仍值得继续细化的问题。

## 1. 产品定位

### 已确认

第一版按 `面向健身房 / 工作室的会员服务方案` 设计。

补充约束：

- 平台用户主要是机构会员
- 平台教练是机构内真人教练
- AI 是辅助服务系统，不替代真人教练

## 2. 第一版是否包含饮食模块

### 已确认

参考文章里把训练与饮食放在一起，但你当前重点更偏训练场景。  
第一版是否要把饮食作为独立模块来做？

第一版不做独立饮食 Agent，只在计划生成里保留简化饮食建议。  
后续版本需要把饮食能力单独做成模块。

## 3. 用户注册后的必填字段

### 已确认

注册后我们到底要收集多深的用户数据？

第一版建议只收集影响计划生成的必要字段：

- 年龄
- 性别
- 身高体重
- 目标
- 每周训练天数
- 每次训练时长
- 器械条件
- 训练经验
- 伤病/禁忌

把复杂字段先降级为可选项，避免首次体验太重。

## 4. 穿戴设备权限要做到什么程度

### 已确认

你已经说明这部分现在可以先做空壳。  
那这个空壳要展示到什么程度？

第一版只需要：

- 在注册或画像页展示“连接穿戴设备”的入口
- 显示“暂未接入，敬请期待”或 mock 状态
- 在用户数据结构中预留权限字段

不建议为了 demo 去做伪接入流程，性价比不高。

## 5. 动作指导的知识库来源

### 已确认

第一版知识库先从少量高质量内容开始，不追求数量。  
目前确认的知识库板块包括：

- 动作规范
- 安全注意事项
- 营养常识

后续可以再扩展动作数量和文档深度。

## 6. 高风险问题的处理边界

### 已确认

第一版建议严格保守：

- 不给诊断结论
- 不给康复治疗建议
- 不给继续硬练的建议
- 强提示停止训练、降低强度或寻求专业帮助
- 必要时提示转人工教练

## 7. 管理员后台做到多深

### 已确认

建议做成 `半真实后台`：

- 页面和数据流是真实的
- 知识库状态、模型配置、Prompt 模板可以真实保存
- 但不必一开始支持复杂权限与完整审核流

这样既能演示“可运营”，也不会把后台做成空壳。

## 8. 是否保留会话式聊天页

### 继续沿用

这个产品更像“任务页驱动”，还是“聊天页驱动”？

建议保留聊天能力，但不要以纯聊天为主入口。  
更合理的方式是：

- 主入口是任务型页面
- 页面里嵌入对话助手
- 对话用于补充信息、解释结果、触发任务

这样更符合健身场景，也更利于展示结构化结果。

## 9. 计划结果的展示形式

### 继续沿用

计划应该以聊天消息展示，还是独立卡片/表格展示？

建议以 `卡片 + 表格 + 版本记录` 为主。  
聊天消息只保留“生成过程解释”和“补充说明”。

这是参考文章里“结构化计划”的关键价值点，也最适合你说的“持续显示在个人页面中”。

## 10. 第一版技术路线

### 继续沿用

第一版我们是否需要直接接复杂多 Agent、复杂工作流、复杂 RAG 平台？

不需要。  
更合理的路线是：

1. 先定义清楚 schema 和业务流程
2. 用一个后端编排层承接计划、指导、复盘三类任务
3. 对动作指导优先加 RAG
4. 后面再逐步增强多 Agent 和工作流细节

## 11. Demo 最重要的说服点

### 已确认方向

这版最值得强调的卖点应是：

1. 不是闲聊，而是能形成持续计划闭环
2. 动作指导不是瞎编，而是带引用、可解释
3. 后台可以配置知识库和模型，具备运营能力
4. AI 能与真人教练协同，适合健身房/工作室落地

## 仍建议继续细化的问题

虽然大的方向已经确定，但进入开发前，后面仍建议继续明确：

1. 教练在后台能否直接编辑 AI 计划，还是只允许追加备注
2. 会员是否按门店、教练分组展示
3. 营养常识在第一版里是仅用于引用，还是参与计划生成
4. 是否需要区分“管理员”和“教练”两类后台权限
